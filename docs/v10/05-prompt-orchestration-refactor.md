# v10 提示词与编排重构（参考 oh-my-opencode）

## 目标
- 引入「规划 → 评审 → 执行」的分层协作思路，降低主控认知负担。
- 让提示词更结构化、可复用，并显式列出工具目录与禁用项。
- 强化“委派后不重复执行”的并行协作规则。

## 关键改动

### 1) 分层协作角色
- **planner**：只读规划角色，输出 `docs/plans/<task>.md`，提供里程碑、节点拆解与验收标准。
- **reviewer**：只读审查角色，输出 `docs/reviews/<nodeId>.md`，检查目标、风险、验收与测试策略。
- **oracle**：只读顾问角色，提供关键推理/架构建议，不修改实现。

### 2) 主控提示词增强
- 引入角色速览与分层编排指南（planner/reviewer/oracle）。
- 增加硬性禁令（如需求未更新不得 submit_plan）。
- 增加“委派后不重复执行”的并行协作规则。

### 3) 工具目录可见化
- 主控/工人系统提示追加“工具目录”清单，按类别展示可用工具。
- 通过工具目录帮助代理快速选择正确工具并降低误调用。

### 4) 模块子主控提示词升级
- 增加分层协作提示：不清晰先规划、关键决策先审查，再进入执行拆解。
- 结案收口时引用审查意见，统一风险说明。

## 使用建议
1. **复杂任务**：主控先派 planner 完成计划文档，再派 reviewer 校验关键节点后提交 DAG。
2. **高风险设计**：在 submit_plan 前派 reviewer 评审，避免返工。
3. **疑难问题**：派 oracle 获取高质量推理建议，主控吸收后再决策。

## 对齐原则
- 所有计划与审查必须与 `docs/REQUIREMENTS.md` 一致。
- 变更需求先更新文档，再提交计划或派工。
