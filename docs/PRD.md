## 1. 产品愿景

### 1.1 核心定位

Agent Team 是一个**以任务为中心的多智能体协作系统**，通过定义不同的专家角色来自动化完成软件开发的全生命周期任务。

### 1.2 价值主张

| 价值点 | 说明 |
|--------|------|
| **任务驱动** | 一切活动围绕任务展开，清晰可追溯 |
| **智能协作** | 多角色智能体并行协作，提升效率 |
| **全程可视** | 执行过程、日志、成品全程可见 |
| **即开即用** | 内置角色和工具，无需复杂配置 |

### 1.3 目标用户

- 个人开发者
- 技术专家
- 独立创作者

## 2. 核心理念（第一原则）

### 2.1 任务即中心

系统所有功能必须围绕**任务**展开，遵循以下流程：

```
用户创建任务
    ↓
任务智能拆分
    ↓
多智能体协作执行
    ↓
执行过程记录日志
    ↓
产出成品交付
```

**原则**: 任何功能设计必须回答"这与任务有什么关系？"

### 2.2 四大核心维度

所有功能必须服务于以下四个维度：

| 维度 | 定义 | 验收标准 |
|------|------|----------|
| **执行过程** | 任务如何被执行 | 每个步骤都有日志记录 |
| **任务拆分** | 复杂任务如何分解 | 支持子任务和依赖关系 |
| **日志体现** | 全程可追溯 | 时间线视图完整 |
| **成品体现** | 任务产出什么 | 代码/文档自动关联 |

## 3. 任务模型规范

### 3.1 任务数据结构（强制）

```typescript
interface Task {
  // ========== 基础信息（必填）==========
  id: string;                    // 全局唯一标识，格式：task-{uuid}
  title: string;                 // 任务标题，不超过100字符
  description: string;           // 任务详细描述
  status: TaskStatus;            // 当前状态

  // ========== 执行配置（必填）==========
  role: string;                  // 执行角色标识符
  parentId?: string;             // 父任务ID（子任务必填）
  dependencies?: string[];       // 依赖的任务ID列表

  // ========== 时间追踪（自动）==========
  createdAt: Date;               // 创建时间
  startedAt?: Date;              // 开始执行时间
  completedAt?: Date;            // 完成时间

  // ========== 执行结果（运行时）==========
  artifacts: Artifact[];         // 产出的成品列表
  logs: LogEntry[];              // 执行日志列表
}

type TaskStatus =
  | 'pending'      // 待执行：任务已创建，等待调度
  | 'running'      // 执行中：正在执行
  | 'paused'       // 已暂停：用户手动暂停
  | 'completed'    // 已完成：成功完成
  | 'failed';      // 失败：执行失败，可重试
```

### 3.2 任务生命周期（强制）

```
                    ┌─────────────┐
                    │   pending   │
                    └──────┬──────┘
                           │ 开始执行
                           ▼
                    ┌─────────────┐
              ┌────│   running   │────┐
              │    └──────┬──────┘    │
              │ 暂停      │      完成  │
              ▼           │           ▼
        ┌─────────┐       │    ┌───────────┐
        │ paused  │───────┘    │ completed │
        └────┬────┘ 恢复       └───────────┘
             │
             └──────────────────┐
                                ▼
                         ┌───────────┐
                         │   failed  │
                         └─────┬─────┘
                               │ 重试
                               └───────▶ pending
```

**状态转换规则**:
- `pending` → `running`: 调度器分配资源开始执行
- `running` → `paused`: 用户手动暂停
- `running` → `completed`: 执行成功完成
- `running` → `failed`: 执行出错
- `paused` → `running`: 用户恢复执行
- `failed` → `pending`: 用户重试任务

### 3.3 任务命名规范

| 类型 | 命名格式 | 示例 |
|------|----------|------|
| 主任务 | {动词}{名词} | 开发用户登录功能 |
| 子任务 | [{父任务}] {动词}{名词} | [开发用户登录功能] 设计数据模型 |

## 4. 执行过程规范

### 4.1 标准执行流程（强制）

所有任务执行必须遵循以下7步流程：

```
┌─────────────────────────────────────────────────────────────┐
│ Step 1: 任务接收                                             │
│ - 解析任务描述，识别用户意图                                  │
│ - 验证任务参数合法性                                          │
│ - 输出: 任务意图识别结果                                      │
├─────────────────────────────────────────────────────────────┤
│ Step 2: 任务分析                                             │
│ - 评估任务复杂度（简单/中等/复杂）                            │
│ - 决定是否需要拆分                                            │
│ - 输出: 复杂度评估报告                                        │
├─────────────────────────────────────────────────────────────┤
│ Step 3: 任务拆分（条件执行）                                  │
│ - IF 复杂度 >= 中等 THEN 拆分为子任务                         │
│ - 建立子任务依赖关系                                          │
│ - 输出: 子任务列表 + 依赖图                                   │
├─────────────────────────────────────────────────────────────┤
│ Step 4: 任务分配                                             │
│ - 根据任务类型匹配角色                                        │
│ - 创建智能体实例                                              │
│ - 输出: 智能体分配结果                                        │
├─────────────────────────────────────────────────────────────┤
│ Step 5: 并行执行                                             │
│ - 无依赖的子任务同时执行                                      │
│ - 有依赖的任务按顺序执行                                      │
│ - 输出: 执行进度更新                                          │
├─────────────────────────────────────────────────────────────┤
│ Step 6: 结果汇总                                             │
│ - 收集各子任务执行结果                                        │
│ - 整合输出                                                    │
│ - 输出: 汇总报告                                              │
├─────────────────────────────────────────────────────────────┤
│ Step 7: 成品交付                                             │
│ - 收集所有产出文件                                            │
│ - 关联到任务                                                  │
│ - 输出: 成品清单                                              │
└─────────────────────────────────────────────────────────────┘
```

### 4.2 执行状态追踪（强制）

每个任务执行时必须实时记录以下信息：

| 追踪项 | 说明 | 记录时机 |
|--------|------|----------|
| 当前步骤 | 正在执行的具体步骤 | 步骤切换时 |
| 已用时间 | 从start到当前的时间 | 每分钟更新 |
| Token消耗 | LLM调用成本 | 每次LLM调用后 |
| 工具调用 | 使用了哪些工具 | 工具调用前后 |
| 中间产物 | 执行过程中的临时文件 | 文件生成时 |

## 5. 任务拆分规范

### 5.1 拆分触发条件

| 复杂度 | 判断标准 | 拆分策略 |
|--------|----------|----------|
| 简单 | 单文件修改，< 50行 | 不拆分 |
| 中等 | 多文件修改，50-200行 | 拆分为2-3个子任务 |
| 复杂 | 模块级改动，> 200行 | 拆分为4-6个子任务 |

### 5.2 拆分策略（强制）

系统必须支持以下三种拆分策略：

**策略1: 按角色拆分**
- 适用场景：需要多角色协作的任务
- 示例：产品分析 → 架构设计 → 编码实现

**策略2: 按模块拆分**
- 适用场景：功能独立的模块
- 示例：用户模块 + 订单模块 + 支付模块

**策略3: 按步骤拆分**
- 适用场景：有明确先后顺序
- 示例：设计数据库 → 写API → 写前端

### 5.3 子任务依赖规范

```typescript
interface TaskDependency {
  taskId: string;              // 依赖的任务ID
  type: 'finish-to-start'      // 默认：任务A完成后任务B开始
       | 'finish-to-finish'     // 任务A完成后任务B完成
       | 'start-to-start';      // 任务A开始后任务B开始
}
```

**依赖规则**:
- 不允许循环依赖
- 单个任务最多依赖5个其他任务
- 依赖关系在任务创建时确定，执行期间不变

## 6. 日志规范

### 6.1 日志类型（强制）

| 类型 | 记录内容 | 用途 | 保留期限 |
|------|----------|------|----------|
| 系统日志 | 任务状态变更、调度决策 | 问题排查 | 30天 |
| 执行日志 | 智能体思考、工具调用 | 过程追溯 | 90天 |
| 操作日志 | 用户操作、配置变更 | 审计追踪 | 180天 |

### 6.2 日志数据结构（强制）

```typescript
interface LogEntry {
  // 基础信息
  id: string;                    // 日志唯一ID
  timestamp: Date;               // 时间戳（UTC）
  level: 'debug' | 'info' | 'warn' | 'error';

  // 关联信息
  taskId: string;                // 关联任务ID
  agentId?: string;              // 关联智能体ID
  userId?: string;               // 操作用户ID

  // 日志内容
  type: 'thought'      // 智能体思考过程
      | 'action'       // 执行动作
      | 'tool_call'    // 工具调用
      | 'tool_result'  // 工具结果
      | 'milestone'    // 里程碑
      | 'status_change'// 状态变更
      | 'error';       // 错误

  content: string;               // 日志内容
  metadata?: Record<string, any>;// 扩展信息
}
```

### 6.3 时间线视图规范

任务详情页必须提供时间线视图，格式如下：

```
[HH:MM:SS] {事件图标} {事件描述}

           {关联链接（可选）}
```

**示例**:
```

[10:00:05] 🔍 开始分析任务
```

## 7. 成品规范

### 7.1 成品类型（强制）

```typescript
interface Artifact {
  id: string;                    // 成品唯一ID
  taskId: string;                // 关联任务ID

  type: 'code'        // 源代码文件
      | 'document'    // 文档（Markdown、Word等）
      | 'diagram'     // 图表（架构图、流程图）
      | 'test'        // 测试用例/报告
      | 'config'      // 配置文件
      | 'data';       // 数据文件（JSON、CSV等）

  name: string;                  // 文件名
  path: string;                  // 存储路径（相对路径）
  size: number;                  // 文件大小（字节）
  mimeType: string;              // MIME类型
  createdAt: Date;               // 创建时间
  checksum?: string;             // 文件校验和（SHA256）
}
```

### 7.2 成品管理规范

**自动收集**:
- 智能体创建的文件自动关联到当前任务
- 关联时机：文件写入后5秒内

**版本追踪**:
- 同一文件的多次修改保留历史版本
- 最多保留10个历史版本
- 超出的旧版本自动归档

**快速访问**:
- 任务详情页直接展示所有成品
- 支持按类型筛选
- 支持文件名搜索

**导出打包**:
- 支持将任务成品打包为ZIP下载
- 包内保持原始目录结构

### 7.3 成品命名规范

| 类型 | 命名格式 | 示例 |
|------|----------|------|
| 代码文件 | {模块}-{功能}.{扩展名} | user-login.ts |
| 文档 | {日期}-{标题}.md | 20260220-架构设计.md |
| 图表 | {主题}-{类型}.png | 系统架构-流程图.png |
| 测试 | {模块}.test.{扩展名} | user-login.test.ts |

## 8. 角色系统规范

### 8.1 内置角色（强制）

| 角色标识 | 角色名称 | 职责 | 典型任务 |
|----------|----------|------|----------|
| task-analyzer | 任务分析师 | 分析任务、决定拆分策略 | 所有任务的入口 |
| product-manager | 产品经理 | 需求分析、PRD编写 | 需求相关任务 |
| architect | 架构师 | 技术方案、架构设计 | 设计相关任务 |
| backend-dev | 后端开发 | API开发、业务逻辑 | 后端编码任务 |
| frontend-dev | 前端开发 | UI实现、组件开发 | 前端编码任务 |
| tester | 测试工程师 | 测试计划、用例执行 | 测试相关任务 |

### 8.2 角色匹配规则（强制）

```typescript
function matchRole(task: Task): string {
  const desc = task.description.toLowerCase();

  // 需求相关
  if (desc.includes('需求') || desc.includes('prd') || desc.includes('产品')) {
    return 'product-manager';
  }

  // 架构相关
  if (desc.includes('架构') || desc.includes('设计') || desc.includes('技术方案')) {
    return 'architect';
  }

  // 前端相关
  if (desc.includes('前端') || desc.includes('ui') || desc.includes('界面')) {
    return 'frontend-dev';
  }

  // 后端相关
  if (desc.includes('后端') || desc.includes('api') || desc.includes('接口')) {
    return 'backend-dev';
  }

  // 测试相关
  if (desc.includes('测试') || desc.includes('用例') || desc.includes('质量')) {
    return 'tester';
  }

  // 默认角色
  return 'task-analyzer';
}
```

## 9. API 规范

```typescript
POST /api/tasks

Response: Task

// 获取任务执行日志

Response: LogEntry[]

// 获取任务成品
GET /api/tasks/:id/artifacts
Response: Artifact[]

// 列出子任务
GET /api/tasks/:id/subtasks
Response: Task[]

// 重试失败任务
POST /api/tasks/:id/retry
Response: { "status": "pending" }

// 暂停任务
POST /api/tasks/:id/pause
Response: { "status": "paused" }

// 恢复任务
POST /api/tasks/:id/resume
Response: { "status": "running" }
```

## 10. 数据存储规范

```
data/
├── tasks/                      # 任务数据
│   ├── task-{id}.json          # 任务元数据
│   └── index.json              # 任务索引
```

### 10.2 持久化策略（强制）

| 数据类型 | 持久化策略 | 说明 |
|----------|------------|------|
| 任务数据 | 立即持久化 | 任何变更立即写入磁盘 |

| 成品文件 | 立即持久化 | 写入时立即fsync |
| 缓存数据 | 不持久化 | 重启后清空 |

## 12. 里程碑与验收

### 12.1 开发里程碑

#### Phase 1: 任务核心（Week 1-2）

| 功能 | 验收标准 | 优先级 |
|------|----------|--------|
| 任务CRUD | 接口响应<100ms，数据正确持久化 | P0 |
| 任务状态机 | 状态转换正确，事件触发正常 | P0 |
| 基础执行引擎 | 简单任务60秒内完成 | P0 |
| 执行日志 | 每个步骤都有日志，时间线完整 | P0 |

#### Phase 2: 任务拆分（Week 3）

| 功能 | 验收标准 | 优先级 |
|------|----------|--------|
| 智能任务拆分 | 复杂任务拆分为2-5个子任务 | P0 |
| 子任务依赖 | 依赖关系正确，无循环依赖 | P0 |
| 并行执行 | 无依赖任务同时执行 | P0 |
| 结果汇总 | 子任务结果正确汇总 | P1 |

#### Phase 3: 成品与展示（Week 4）
