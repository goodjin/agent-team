# PRD Builder Skill

基于**对标法**的产品需求文档生成器，通过行业研究和竞品分析，一次性生成高质量、AI友好的PRD文档。

## 核心理念

### 对标法（Benchmarking）

**核心思想**：不从头描述产品，而是基于现有竞品来表达需求。

**公式**：`产品 = 对标物 + 增量描述`

**示例**：
- "我要做一个瑞幸咖啡小程序，但要增加会员积分功能"
- "类似Notion，但要支持实时协作"
- "参考微信支付，但手续费更低"

### 高度信息压缩

用一个对标物名称 + 增量描述，传递大量信息：
- 不需要解释"什么是点单系统"
- 不需要描述"购物车的基本功能"
- AI可以直接基于对标物展开细节

### AI友好原则

- **一次性补齐信息**：不让用户回答无法回答的问题
- **基于已知参考**：利用竞品的已知信息填补空白
- **结构化输出**：生成可直接用于下游的标准化文档

---

## 触发条件

用户输入包含以下内容时自动触发：
- "需要开发"、"想要实现"、"做一个"
- "增加一个功能"、"修改一个功能"
- "帮我设计"、"生成PRD"
- 或任何需求相关的描述

## 工作流程

### 阶段 1: 需求理解与对标物识别

#### 1.1 识别对标物

**执行步骤**:
1. 分析用户描述，识别提到的产品/系统名称
2. 如果用户明确提到对标物（如"类似瑞幸"），直接使用
3. 如果没有明确对标物，询问用户选择参考对象

**询问模板**:
```
我理解您想要开发 [需求概述]。

为了更好地理解您的需求，请选择一个或多个参考产品：
1. [竞品1] - [简述]
2. [竞品2] - [简述]
3. [竞品3] - [简述]
4. 其他（请描述）

您的产品与这些竞品的主要差异是什么？
```

#### 1.2 提取增量需求

**执行步骤**:
1. 识别用户描述中的"增量"部分（要做但竞品没有的）
2. 识别用户描述中的"减量"部分（竞品有但不要的）
3. 识别用户描述中的"修改"部分（与竞品不同的实现方式）

**增量需求模板**:
```
| 类型 | 描述 | 对标物对比 |
|-----|------|----------|
| 增量 | [新增功能] | 对标物没有 |
| 减量 | [不需要的功能] | 对标物有，但我们不需要 |
| 修改 | [差异化实现] | 对标物是X，我们要Y |
```

---

### 阶段 2: 行业研究与竞品分析

#### 2.1 竞品功能模块分析

**执行步骤**:
1. 列出每个对标物的核心功能模块
2. 提取每个模块的关键特性
3. 对比不同竞品的实现差异

**输出格式**:

```markdown
## 竞品功能模块对比

### 模块1: [模块名称]

| 竞品 | 功能特性 | 用户体验 | 技术实现 |
|-----|---------|---------|---------|
| [竞品1] | [特性描述] | [体验评价] | [实现方式推测] |
| [竞品2] | [特性描述] | [体验评价] | [实现方式推测] |
| **我们的方案** | [差异化特性] | [目标体验] | [计划实现] |

### 模块2: [模块名称]
...
```

#### 2.2 竞品对比矩阵

**输出格式**:

```markdown
## 竞品对比矩阵

| 维度 | 竞品1 | 竞品2 | 竞品3 | 我们 |
|-----|-------|-------|-------|-----|
| **核心功能** |
| 功能A | ✓ | ✓ | ✗ | ✓（增强版）|
| 功能B | ✓ | ✗ | ✓ | ✓ |
| **用户体验** |
| 易用性 | ⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ | 目标⭐⭐⭐⭐⭐ |
| 性能 | 快 | 中等 | 快 | 目标快 |
| **商业模式** |
| 定价 | 免费+付费 | 订阅 | 免费 | [我们的模式] |
```

#### 2.3 最佳实践建议

基于竞品分析，输出建议：

```markdown
## 最佳实践建议

### 推荐采用
1. **[功能A]**: 竞品1和竞品2都有此功能，建议实现（来源：行业标配）
2. **[交互模式B]**: 竞品3的实现体验最好，建议参考

### 建议避免
1. **[功能C]**: 竞品都有但用户反馈不好，建议不实现
2. **[设计D]**: 过于复杂，建议简化

### 创新机会
1. **[功能E]**: 竞品都没有，但是[原因]，建议作为差异化亮点
2. **[体验F]**: 当前竞品的痛点是[痛点]，我们可以[方案]
```

## PRD文档质量标准

### 需求描述规范

良好的需求描述应做到精确、无歧义、可执行。生成PRD时，遵循以下规则：

**1. 避免模糊量词，使用具体数值**
- ❌ "一段时间后订单自动取消"
- ✅ "30分钟后订单自动取消"

**2. 避免模糊指代，使用明确主语**
- ❌ "支付后它会被取消"
- ✅ "支付超时后，订单会被取消"

**3. 避免隐含条件，显式列出所有前提**
- ❌ "支付成功后更新状态"
- ✅ "当订单处于已创建状态且支付成功时，订单状态变为已支付"

**4. 避免多义句，一句话只表达一个规则**
- ❌ "用户取消订单并退款"
- ✅ "用户取消订单后，系统发起退款（仅当订单已支付时）"

**5. 一句一规则，复合逻辑拆行书写**
- ❌ "如果a且b，则c，否则如果d则e"
- ✅ "如果a且b，则c。\n如果d，则e。"

**6. 使用显式主语，明确操作主体**
- ❌ "状态变为已完成"
- ✅ "系统将订单状态变为已完成" / "用户提交后，系统..."

**7. 使用标准句式**
- 条件句：`当 [条件] 时，[结论]`
- 规则句：`如果 [条件]，则 [结论]`
- 操作句：`[主体] 执行 [操作]，[结果]`

**8. 澄清关键定义（有则写，无则不强求）**
- 实体定义：描述核心数据对象的含义（如"订单 = 用户提交的一次购买请求"）
- 状态定义：列出实体的所有可能状态及其含义
- 规则定义：明确业务规则的触发条件和结果
- 事件定义：列出关键领域事件（如"订单已支付"、"库存不足"）
- 示例：对复杂逻辑提供具体示例帮助理解

**9. 用户交互必须穷举所有对象分支**
- 同一操作（如双击）针对不同目标（空白区域 / 已有元素 / 边界）时，行为可能不同，必须各自写一条独立的 AC
- ❌ "双击可创建或编辑注释"
- ✅ AC-006-01: "双击空白区域 → 在该位置创建新便签"；AC-006-02: "双击已有高亮 → 打开该高亮的注释编辑框"

**10. 同一功能的所有 UI 入口必须各自写独立 US**
- 若同一功能可通过多个入口触发（如菜单 + 启动页 + 快捷键），每个入口必须有独立的用户故事和 AC
- ❌ "用户可通过菜单或启动页打开最近文件"（合并为一条 US）
- ✅ US-015-A: "用户在菜单 > 文件 > 最近打开 中选择文件"；US-015-B: "用户在启动页列表中点击最近文件"

**11. 性能指标必须包含设备规格和验证工具**
- 不写主观词（"流畅"、"快速"、"秒开"）；必须写成可测格式
- ❌ "滚动时保持流畅"
- ✅ "在搭载 Apple M1 及同等性能设备上，滚动帧率 ≥ 60fps，使用 Instruments > Core Animation 验证"

## 注意事项

1. **对标法优先**: 始终先识别对标物，基于对标物展开需求
2. **一次性完整**: 不要让用户反复补充信息，基于竞品分析自动补全
3. **严格编号**: 所有功能、故事、验收标准都必须有唯一编号
4. **附录完整**: 竞品分析作为附录，提供完整的参考信息
5. **AI友好**: 生成的文档要便于后续架构设计Skill直接使用
